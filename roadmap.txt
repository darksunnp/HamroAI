## Phase 2: Data Cleaning & Preprocessing (Week 5–8)

### 2.1 — Text Cleaning Pipeline
Build a sequential cleaning pipeline. Process every document through these steps **in order:**

1. **Encoding normalization** — Convert everything to UTF-8. Nepali Devanagari has Unicode normalization issues (NFC vs NFD). Standardize to NFC.
2. **HTML/boilerplate removal** — Strip all HTML tags, navigation menus, ads, cookie notices, footers.
3. **Language identification** — Use `fasttext` language ID model (`lid.176.bin`) to filter out non-Nepali documents that slipped in. Keep documents with >80% Nepali confidence. For code-switched text, you'll need a more nuanced filter (keep documents identified as Nepali OR Hindi with Nepali script present).
4. **Deduplication** — This is critical. Use **MinHash + LSH** (via the `datasketch` library) to find and remove near-duplicate documents. News sites republish the same story across portals constantly. Exact dedup first (hash-based), then fuzzy dedup.
5. **Quality filtering:**
   - Remove documents shorter than 50 characters.
   - Remove documents that are mostly numbers/punctuation.
   - Remove documents with excessive repetition (same sentence repeated).
   - Remove documents that look auto-generated or are just lists of links.
6. **PII scrubbing** — Remove phone numbers, email addresses, and Nepali-format personal identifiers using regex patterns.
7. **Script validation** — Ensure Devanagari text uses correct Unicode code points (Nepali Devanagari range: U+0900 to U+097F). Remove documents with corrupted/garbled Unicode.

### 2.2 — Data Organization
After cleaning:
- Store cleaned text as **JSONL files** (one JSON object per document, with fields: `text`, `source`, `url`, `date_scraped`, `language`, `domain`).
- Split by domain: `news`, `legal`, `literature`, `social`, `wikipedia`, `government`.
- Calculate and log statistics: total tokens, unique documents, average document length, domain distribution.
- **Target after cleaning:** 2–5 GB of high-quality, deduplicated Nepali text.

## Phase 3: Custom Nepali Tokenizer (Week 7–9)

### 3.1 — Why This Matters
To illustrate the problem: the word "नेपाल" (Nepal) might take 1 token in a Nepali-optimized tokenizer but 4–6 tokens in Llama's default tokenizer. This means:
- **3–5x higher inference cost** for Nepali text vs. English.
- **Slower generation speed.**
- **Reduced effective context window** (your 8K context window becomes effectively 2K for Nepali).

### 3.2 — Build the Tokenizer
1. **Gather your tokenizer training corpus** — Use your entire cleaned text corpus from Phase 2. The more representative, the better.
2. **Train a BPE tokenizer** using HuggingFace's `tokenizers` library:
   - Vocabulary size: Start with **32,000 tokens**. This is a good balance.
   - Include all Nepali Devanagari characters, conjuncts (half-letters), and common ligatures as base characters.
   - Include English characters and digits (for code-switching support).
   - Add special tokens: `<s>`, `</s>`, `<pad>`, `<unk>`, and any others your base model uses.
3. **Train an alternative SentencePiece (Unigram) tokenizer** for comparison.
4. **Measure fertility** — Fertility = (number of tokens produced) / (number of words in the text). Lower is better.
   - Your custom tokenizer should have a fertility of ~1.2–1.5 for Nepali.
   - Compare against Llama 3's tokenizer (which will likely show fertility of 3.0–5.0 for Nepali).
   - Document these numbers — this is one of your key technical results.

### 3.3 — Tokenizer Integration Strategy
You have two paths for using this tokenizer with a base model:

- **Path A (Simpler):** Use the base model's original tokenizer but do continual pre-training to improve its Nepali representations. The tokenizer stays inefficient, but the model learns Nepali within its existing token space.
- **Path B (Better but harder):** Extend the base model's tokenizer vocabulary by merging your Nepali tokens into it, then resize the model's embedding layer and do continual pre-training. This is what Sarvam did for Hindi.

**Recommendation:** Start with Path A to get a working model fast. Then attempt Path B as an improvement.
